# Contact & Enquiry Architecture — V15

## Objective

Turn interest generated by destinations, journey styles, canonical products, the Trip Finder and editorial pages into a qualified, privacy-conscious enquiry that reaches the Sawla Tours team reliably.

## Canonical conversion route

- Public planning page: `/enquire`
- Confirmation page: `/enquire/thank-you`
- Submission endpoint: `/api/enquire`
- Retired contact URLs: `/contact-us` and `/contact` permanently redirect to `/enquire`

The confirmation page is `noindex` and is intentionally excluded from the sitemap.

## Conversion design decision

V15 uses one progressive, single-page form rather than a long multi-screen wizard. High-value private travel benefits from useful planning detail, but international visitors also need a form that remains understandable, keyboard-accessible and resilient on slower connections.

The form is divided into three visible sections:

1. Contact details
2. Journey basics
3. Helpful preferences

Only the information needed to begin a useful response is required. Phone/WhatsApp, country, interests, budget and planning considerations remain optional.

## Required visitor fields

- full name;
- valid email;
- travel month, dates or a clear “flexible” indication;
- approximate duration;
- at least one adult traveller;
- a short journey description;
- privacy acknowledgement.

WhatsApp becomes required only when the visitor selects WhatsApp as the preferred contact method.

## Journey context

Canonical product CTAs already use `?journey=<slug>`. V15 resolves that slug against the authoritative 25-product dataset and displays the selected journey above the form. The API re-resolves the slug rather than trusting the browser-supplied title.

## Trip Finder handoff

Trip Finder answers now carry into `/enquire` through non-personal query parameters:

- travel window;
- duration;
- interests;
- group size;
- budget;
- source attribution.

The visitor still adds contact details and a message before submission. Trip Finder answers are no longer discarded when moving to the full form.

## Submission contract

The API accepts:

- `application/json` from the interactive form;
- `multipart/form-data`;
- `application/x-www-form-urlencoded`.

This corrects the previous contract mismatch where the browser posted form data but the API attempted to parse JSON only.

## Delivery sequence

1. Validate and normalize the payload on the server.
2. Confirm the referenced canonical journey.
3. Generate a non-personal enquiry reference.
4. Deliver the full team email.
5. Only after team delivery succeeds, return success to the browser.
6. Attempt the traveller acknowledgement email.
7. Send the optional CRM/workflow webhook when configured.
8. Redirect the browser to the confirmation page.

A failed traveller acknowledgement does not conceal a successfully delivered team enquiry. A failed team delivery does not produce a false success message.

## Team email

The team notification contains:

- reference;
- traveller contact details;
- preferred contact method;
- date and group information;
- selected canonical journey;
- interests, pace, style and budget signals;
- message and practical considerations;
- source, landing page and UTM fields;
- submission time.

The email `Reply-To` is the traveller’s validated email address.

## Traveller acknowledgement

The acknowledgement includes:

- enquiry reference;
- concise trip summary;
- selected journey when relevant;
- realistic response wording;
- a warning not to send passport, card or sensitive medical information at the initial stage.

## Security and abuse controls

V15 includes layered baseline controls:

- 64 KB request-body ceiling;
- structured server-side allowlists;
- length and range checks;
- output encoding for HTML email;
- honeypot field;
- minimum completion time;
- excessive-link spam check;
- cross-site browser submission rejection;
- process-local request throttling;
- idempotency keys for both emails;
- no client-supplied canonical journey title is trusted;
- no secrets exposed to browser code.

The included rate limiter is an application-instance baseline. High-volume production infrastructure should additionally apply a durable or platform-level rate limit. CAPTCHA should be added only if observed abuse justifies its conversion and privacy cost.

## Privacy choices

The form does not request:

- passport or identity copies;
- card or bank details;
- sensitive medical records;
- newsletter consent;
- file uploads.

The privacy policy now discloses enquiry-source metadata, anti-spam information and optional CRM/workflow processing.

## Analytics

The confirmation route pushes a `generate_lead` event to `window.dataLayer` when a valid reference is present. It contains only:

- the non-personal reference;
- lead type.

No traveller name, email, phone, dates or message is sent to the data layer by this component.

## WhatsApp

The floating WhatsApp button now includes the current Sawla Tours page URL in the message. This gives the team useful context without creating hidden tracking or requiring visitors to repeat which page they were viewing.
